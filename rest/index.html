Microsserviços e REST

O que podemos aprender com eles?

Monolítico x Microsserviço

Monolítico
  • Aplicação única
  • Frontend integrado à App
  • Novas funcionalidades são acrescidas na mesma App
  • Cada vez que for acessar dados de outras Apps, é necessário conectar-se a um novo banco de dados ou duplicar esses dados
  • Em caso de conexão ao banco de dados, a aplicação não necessariamente segue as regras de negócio pertinentes ao banco de dados
  • Desenvolvimento somente em uma linguagem
Microsserviço
  • Aplicação composta de um conjunto de aplicações menores
  • Frontend pode ser desenvolvido fora das aplicações menores
  • Cada vez que haja a necessidade de conectar-se a novos dados, usa-se outra aplicação para a conexão, respeitando as regras de negócios e evitando conectar-se diretamente a outros DBs
  • Abstração da linguagem de programação ao criar novos microsserviços
  • Aplicação única

O que é REST

Apresentado em 2000 pela primeira vez, REpresentational State Transfer é um estilo de arquitetura para sistemas distribuídos de hipermídia

Ele é guiado por 6 princípios. São eles:

  • Cliente-Servidor - Ao separar as preocupações de interface de usuário das preocupações de armazenamento de dados, torna-se possível a portabilidade da interface de usuário através de múltiplas plataformas e da escalabilidade da aplicação ao simplificar os componentes do servidor
  • Stateless - Cada requisição do cliente para o servidor deve conter todas as informações ncessárias para compreender a requisição e não pode tomar vantagem de qualquer contexto armazenado no servidor. Os dados de sessão são portanto armazenados diretamente com o cliente.
  • Cacheável - Limitações de cache obrigam que o dado dentro de uma resposta à requisição seja implicitamente ou explicitamente rotulados como cacheavel ou não-cacheavel. Se uma resposta é cacheavel, então o cache do clietne tem direito de reusar os dado dessa resposta anterior para requisições posteriores e equivalentes.
  • Interface Uniforme - Uma vez projetada a forma como deve ser acessada a API, deve-se seguir à risca essas regras para manter a estrutura das respostas da API uniforme
  • Sistema em Camadas - O sistema em camadas permite que a arquitetura seja composta de camadas hierárquicas ao delimitar o comportamento do componente de forma que o componente não possa "ver" além da camada imediata com a qual ela está interagindo.
  • Código Sob Demanda - REST permite que a funcionalidade do cliente seja extendida ao baixar e executar código na forma de scripts (JavaScript, por exemplo). Isso simplifica os clientes ao reduzir o número de funcionalidades necessárias para serem pré-implementados.

O que pode ser feito com REST

REST permite desenvolver qualquer aplicação web que tenha todo tipo de operações CRUD. Porém, ao invés de utilizar somente os métodos HTTP GET e POST, é recomendado utilizar cinco métodos, onde cada um é específico para cada tipo de transação

HTTP GET

O método GET é utilizado somente para recuperar dados em modo de leitura

Exemplos de requisições GET

  • HTTP GET http://www.appdomain.com/users
  • HTTP GET http://www.appdomain.com/users?size=20&page=5
  • HTTP GET http://www.appdomain.com/users/123
  • HTTP GET http://www.appdomain.com/users/123/address

HTTP POST

O método POST é utilizado para inserir novos dados através do link do REST.

Exemplos de requisições POST

  • HTTP POST http://www.appdomain.com/users/123
  • HTTP POST http://www.appdomain.com/users/123/accounts/

HTTP PUT

PUT serve primariamente para atualizar um dado existente da aplicação. Caso esse dado não exista, a aplicação pode decidir se cria um novo dado ou não.

Exemplos de requisições PUT

  • HTTP PUT http://www.appdomain.com/users/123
  • HTTP PUT http://www.appdomain.com/users/123/accounts/456

HTTP DELETE

DELETE, como o próprio nome já diz, é utilizado para remover dados.

Exemplos de requisições DELETE

  • HTTP DELETE http://www.appdomain.com/users/123
  • HTTP DELETE http://www.appdomain.com/users/123/accounts/456

HTTP PATCH

Métodos PATCH são utilizados para fazer atualização parcial de dados. O PATCH é mais recomendado para atualização parcial e o PUT deve ser usado apenas na substituição completa dos dados.

No entanto, o PATCH não é compatível com todos os navegadores, nem servidores. IE8, PHP, Tomcat e Django, por exemplo, não tem implementação ou falta suporte para o método.

Como adaptar Monolítico para Microsserviço

A transição de uma App Monolítica para Microsserviços pode ser realizada de duas formas:

  • Abandonar o desenvolvimento da aplicação por completo para poder focar no novo modo de produção
  • Implementar gradualmente microsserviços até que a aplicação principal deixe de existir e passe a ser somente mais um serviço
